N
01 · Need — why we built it
N
Course registration at UTM is manual and stressful

Students plan timetables by hand in Excel and WhatsApp. Nothing checks clashes for them or helps when a section fills up.

  • Manual clash-checking across every offered section
  • Sections fill fast — no backup plan ready
  • Paper handbooks — course data not digital
  • No recovery path when registration partially fails
A
02 · Approach — features & technology
A
Four subsystems. One pipeline.

React SPA → Express.js REST API → MySQL (31 tables). JWT + bcryptjs auth with DB session revocation. Recursive clash-free pattern engine.

  • User Management — auth, profile, dashboard, alerts
  • Handbook & Timetable — Excel upload, course/section CRUD
  • Pattern Generation — clash detection, lecturer preference
  • Student Registration — Draft Vault, compare, recovery
B
03 · Benefit — what students get
B
Hours of planning become seconds. With a safety net.

Every UTM student gets a clash-free plan before registration — and a way back when it breaks.

  • Before — clash-free patterns generated automatically
  • During — Draft Vault (30 slots) + side-by-side compare
  • After — partial recovery, drop simulation, gap filling
  • Admins — handbook & timetable digitized via Excel upload
C
04 · Competition — what makes it unique
C
Other tools stop at a timetable. We handle what happens next.

Generic schedulers output one timetable and stop. CourseBuddy plans, executes, and recovers.
